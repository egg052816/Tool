3. ATS / Terminal 架設環境
ATS & Terminal

ATS / Terminal 操作原則（2 選 1）：

  1. ATS 服務啟動與關閉（Web UI）

    可在 Teminal 下指令控制 Android Test Station 服務的生命週期：

    mtt start（開啟服務並透過網址存取）
    mtt stop（關閉服務）

    ＊ mtt start 跑完需要點擊 FireFox Web Browser 才會跑出 ATS 網頁介面。


  2. Terminal 執行

    若想在 Terminal環境執行 Tradefed，必須確保環境無殘留進程：

    step 1. 必須先關閉 ATS 服務 : mtt stop（關閉ATS）
    step 2. 確保新開 terminal 及 adb 是一個正確的乾淨環境 : adb kill-server

4. Run XTS
Run

測試版本：

  • 原則上，XTS 測試應以客戶自測報告的版本為基準。若測試版本與客戶版本不同（例如客戶使用非標準的 Bundle 或未提供版本），則採用當前最新版本進行執行測試。
  • 查閱完客戶自測報告後，請於 Jira 註明其測試版本，並依該版本執行自動化測試。

執行前檢查：

5. Retry
詳細Retry資訊((超連結))
Retry

Retry 策略分成「非 CTS 測項」與「CTS 測項」兩大類，外加「ToT + Waiver」特別情境。

何謂TOT?

TOT 為 Google 本月更新的程式碼版本，並未經過廣泛測試或正式釋出的最新程式碼，又稱 Daily Build。

如下圖

非 CTS 測項 Retry 原則

除 CTS 外，若依照log顯示錯誤內容 Retry 多次都通過不了，建議使用 ToT（Daily Build）做 Retry 看看測項能否通過，或許是近期 Google 才有更新，正式 release 尚未包含修正。

CTS 測項 Retry 原則

CTS 測項較為複雜，若 Fail 項目已經收斂到一個階段（只剩少數固定 Fail），請優先依照以下順序處理：

  1. 手動更新手機 Mainline 至較新的版本。

    登入Google帳號 -> 點擊system Android版本 -> 點擊Google System Update -> 更新至最新版(通常最新版本為上個月) -> 移除Google帳號

  2. 打開 MCTS,更改 Tradefed (TF) 設定，將動態下載參數改為 True：「 ENABLE_XTS_DYNAMIC_DOWNLOADER=True 」。
  3. 上述都做過、怎麼 Retry 都無法 pass，才改用 ToT（Daily Build）測試。
ToT & Waiver 處理

若用 ToT 單跑測項:
1.跑TOT通過時，記得額外附上單項測試報告。若跑TOT也顯示fail的狀況，代表一開始你Retry方法就使用錯誤，請回測試版本重新測試。
2.發現結果都變成 0，代表 Google 已經將測項移除了，因此不論怎麼測試都不會通過。 發生上述情況時，請：

  • 使用公司帳號登入 foxconn.corp-partner.google.com檢查該測項是否已 waiver。
  • 若測項已經 waiver，提交時記得附上對應的 waiver ID 給 RD，讓 RD / 客戶後續有依據可查。

※ 內部另有整理一份「已確認可用的 Retry 手法與常見 Waiver ID」列表，可視需要補充到本頁或存放在 Waiver 工具中，以利後續查詢((超連結))。

6. Report Check
Report

完成所有測試後，請使用 ReportCheck Tool，做一次 自動 + 手動 的報告檢查，確認以下事項：

  • 自動產生的報告內容是否完整、無明顯錯誤。
  • 測試結果是否與實際 Run 的套件／版本一致。
  • 工具會在本機 tmp 位置自動生成截圖與附件，請一併收集，提供給 RD 做 Review。

最終確認過的報告檔、截圖、附件路徑，整理在 JIRA 上給RD review ，待 PM 提交給 Google 且 auto approved 後測試就完成了 。
