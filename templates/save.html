XTS TEST – FLASH SW FLOW
前情提要
General Rules

✯ 除了 STS 外，所有 XTS 測試項目（CTS / CTS Verifier / GTS / GTS-Interactive / GTS-Root / CTS-on-GSI / VTS…）以及 Pretest，一律先燒錄 User Build SW。

✯ 只有STS 測試會燒 Userdebug SW。

✯ Android 15（A15）以上版本 新增額外側項：

  • A15 以上：需依圖燒錄 GTS-Root 進行測試。
  • A15 以下：不需要 燒錄 GTS-Root，其他流程相同。

User / UserDebug Build
Build Type

進入3pl資料夾後 ，3種zip直接抓下來：



  1. User Build: 以下圖為客戶就有提供windows環境的bat檔 跟 Linux環境的sh檔

  2. UserDebug Build:概念跟User Build一樣，只是要注意彼此不同包。
  3. 客戶自家燒錄Tool及Driver: (Tool 因每一家客戶不同就不多作介紹)
  • 燒錄時，樣機需要進入Fastboot Mode，如果客戶有提供腳本通常可直接執行bat檔；若無則需要手動輸入指令或按下指定按鍵(通常為關機鍵+音量下鍵)
    • 燒錄步驟：
    1. 在 Terminal 輸入 adb reboot bootloader 樣機會進入 Bootloader 界面
    2. 進入第一層後，輸入 fastboot devices 檢查電腦有無driver，如果沒driver，電腦在第一層會找不到樣機型號
      3. 如果找不到型號，打開裝置管理員，你應該能看到一個帶有警示符號的Android裝置

CTS-on-GSI 燒錄流程
CTS-ON-GSI

CTS-on-GSI 需要額外從內網下載 Google GSI，並修改 fastboot bat 檔中的 system image 路徑，步驟如下：

  1. 開啟內網網站：http://10.57.42.97/XTS/ → 進入 VTS 資料夾。

  2. 依照 JiRA 上標示的 Android 版本選擇，例如 Android 12.1：
    VTS-12.1_r14/ → 再進到底下的 R14 子資料夾。

  3. 在資料夾中找到對應架構的 signed GSI_arm64 zip，例如：
    signed-gsi_arm64-img-13267304.zip

  4. 下載並解壓縮後，裡面會有一個 system image:

  5. 將這個 system image 複製到「客戶提供UserBuild」的資料夾中（根據客戶提供的SOP，進行Google System 的燒錄），
可能遇到情形：
  1. 客戶提供燒錄bat檔，就需要將檔案中既有的system.img，改成自己抓對應版本的system.img
  2. 客戶提供自家SOP及燒錄指令
  3. 未提供則在Fastboot Mode 輸入 fastboot flash system system.img，若為lock樣機需去客戶提供資料中找解鎖指令。
VTS 燒錄流程
VTS

VTS會比GSI多燒一個 Vendor-boot-Debug

  1. 客戶一定會需要提供，因此要特別先找vendor_boot-debug.img 放在哪，若客戶沒提供則盡速跟PM反應。
  2. 依客戶SOP為準，若無輸入fastboot flash vendor_boot vendor_boot-debug.img，燒入VTS
